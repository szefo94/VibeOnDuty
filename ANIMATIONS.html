Machine observation

70% of declared clips: dead code at runtime. Clips load fine. tickEnemyAnimation waterfall never reaches them. Not an asset problem — routing problem.

How locomotion is currently decided

Speed = boolean. Moving or not. Attack state moving → run. Not moving → attack. No interpolation. No velocity magnitude. No directionality.

// Current selection logic (simplified) — enemyAnimations.js if (state === 'attack' || state === 'spotted') { clip = isMoving ? 'run' : (muzzleFlashT > 0 ? 'shoot' : 'attack'); } // isMoving = velocity.length() > 0 — there is no speed value, only on/off // result: a soldier going 0.1 m/s and 6 m/s play identical animations

The crossfade problem

crossFadeTo() avoided — freezes incoming clip at frame 0. Fix: fadeOut + reset().fadeIn().play(). Works. Duration = flat constant: 0.22s smooth / 0s snap.

Sprint → crouch: same 0s snap as bullet-death. Shuffle → idle: same 0.22s as full reversal. No pose carry-over. No velocity. No context.

Part II

Where It Breaks

Player-visible failure modes, mapped to code root causes. Ordered by impact.

  • 01
    Walk and run are parallel universes

    0.5 m/s and 6 m/s play same run clip. No speed blend. Low speed = foot-slide. High speed = stride doesn't match velocity.

    High
  • 02
    Strafe is invisible

    Enemies strafe in attack state (enemyStates.js). Slide sideways while playing forward run. Strafe clips declared, loaded, never selected.

    High
  • 03
    Shoot animation is sub-perceptual

    shoot fires only while muzzleFlashT > 0: ~55ms window. Human persistence ~100–150ms. Clip starts, gets crossfaded before eyes register. Player sees only muzzle flash.

    High
  • 04
    No additive layer — everything is a full-weight replacement

    AdditiveAnimationBlendMode unused. Every clip fights for 100% weight. Hit reaction replaces locomotion instead of layering on top. Breathing, aim sway, recoil: no clips at all.

    High
  • 05
    Upper body and lower body are locked together

    No bone mask. Walking + shooting: one clip wins. Locomotion wins. Arm gets nothing — gun floats at fixed pose while legs cycle. Spine split would let legs walk while arms aim independently.

    High
  • 06
    Crossfade ignores context — 0.22s is never the right answer

    Idle → walk: 0.22s feels slow. Sprint → aim: 0.22s feels abrupt. Right duration depends on velocity delta + clip distance. Inertial blending (carry outgoing pose momentum) makes every transition feel right automatically.

    Med
  • 07
    No procedural foot IK

    Ramp tiles raise floor. Feet embed into or float above surface. Two-bone ankle IK + per-foot raycast against collision surface = grounded characters. No custom animation per slope needed.

    Med
  • 08
    Death has no lead-in

    killEnemy snaps to death at 0. Correct. But without preceding hit-stagger additive, enemy goes full-animation → ragdoll-freeze in one frame. Looks like bug, not death.

    Low
  • 09
    Procedural weapon bob competes with GLTF body animation

    1P weapon bob: sine offset on wpn.position. 3P GLTF animates independently. Not synced — weapon floats while body cycles. Shared locomotion phase float would fix it.

    Low
Part III

The Evolution Plan

Five changes. Ship incrementally. Each closes specific gap above. Order matters — each builds on last.

A

Locomotion Blend Tree

locomotion no new assets needed

Replace isMoving boolean with speed float (0–1). Blend idle → walk → run by weight as speed changes. Kills foot-slide. Deceleration feels real.

Wire strafe: dot(velocity, forward). Negative lateral → strafe_l weight, positive → strafe_r. 2D blend tree: forward/back + strafe + speed.

// Proposed: 2D locomotion blend (forward × strafe) const vFwd = dot(velocity, forward); // –1 … 1 const vRight = dot(velocity, right); // –1 … 1 const speed = velocity.length() / MAX_SPEED; actions.idle.weight = Math.max(0, 1 - speed * 2); actions.walk.weight = triangular(speed, 0, 0.5, 1); actions.run.weight = Math.max(0, speed - 0.5) * 2; actions.strafe_l.weight = Math.max(0, -vRight) * speed; actions.strafe_r.weight = Math.max(0, +vRight) * speed; // normalise weights so they sum to 1.0
B

Upper / Lower Body Split via Bone Mask

body masking blender asset needed

AnimationAction supports weight-per-track. Zero spine-up tracks on locomotion layer, zero leg-bone tracks on aim layer. Clean split: legs walk, torso shoots.

Upper body (shoot, reload, attack): arm/spine/head tracks only. Lower body (walk, run, idle): leg tracks only. Quaternius bone names (spine_01, thigh_l/r) already resolved in enemyGLTF.js — same lookup enables masking.

Impact on shoot animation

Shoot clip runs on arm/spine layer only. Legs keep cycling. 55ms muzzle-flash window no longer bottleneck.

C

Additive Layer Stack

additive blending new clips needed

Dedicated additive action group. Each clip type at independent weight, stacks on top of base locomotion:

Breathing

3-sec loop. Spine compression + shoulder rise. Weight 1.0 always, 1.4 during sprint. One Blender export.

Hit Reaction

0.3s directional clips: hit_front/back/left/right. Additive on top of locomotion — stagger without interrupting walk. Weight lerps 1.0→0 over duration.

Aim Offset

Tilts spine + arm toward look target. Driven by pitch. No clip needed — procedural spine rotation applied additively per frame.

Recoil

Currently procedural offset in shoot.js. As additive clip: body rocks on fire, not just camera. recoilT float drives weight.

D

Inertial Blending

transitions no new assets

Capture pose + angular velocity at transition moment. Decay toward incoming clip via critically-damped spring. Transitions feel right regardless of duration — blend curve shaped by physics, not fixed lerp. (Bollo/Ubisoft, standard across AAA.)

Three.js impl: store outgoing bone quaternions + velocities at crossfade time. Apply spring-damper correction offset, decays to zero over ~0.15s. Spring constant controls snappy vs floaty.

// Inertial transition sketch function onTransition(bone, currentQ, nextQ) { const offset = currentQ.clone().multiply(nextQ.conjugate()); const vel = angularVelocity[bone.uuid] ?? new THREE.Quaternion(); inertia[bone.uuid] = { offset, vel, t: 0 }; } function tickInertia(bone, dt) { const s = inertia[bone.uuid]; if (!s) return; // critically-damped spring: x'' = –ω²x – 2ωx' s.t += dt * OMEGA; const decay = (1 + s.t) * Math.exp(-s.t); bone.quaternion.premultiply( slerpQ(identityQ, s.offset, decay) ); }
E

Procedural Foot IK

IK terrain-aware

Two-bone IK per ankle (thigh → calf → foot). Each frame: raycast down from animated foot. Hit above foot → solve IK to plant. Hit below → leave animated pose.

Hip offset: lower pelvis when feet reach down (depression), raise ascending ramp. Characters feel grounded on ramps + column bases. No per-terrain animation work.

BVH already queryable — same system as player movement. Per-foot raycast against wallMeshes returns surface normal for ankle rotation.

Scope note

Most complex step. Ship after A–D stable. Flat maps: zero visual gain. Ramp-heavy maps like Vanguard Complex: significant.

Part IV

Claude × Blender

Claude now reads, writes, and executes bpy scripts inside Blender. Changes asset pipeline for this project.

Claude generates Python: bone by bone, frame by frame, baked to NLA strips for GLB export. Artist job: curate + tune, not keyframe.

What Claude can generate for this project

Additive clip set

bpy script: breathing + hit_front/back/left/right as additive NLA strips on Quaternius rig. Procedural keyframes (spine compression, arm flinch). Exports with ANIM_RETARGET — slots into alias map unchanged.

Locomotion polish pass

Reads existing walk/run clips, generates 6-frame transition poses. Output: walk_to_run + run_to_walk. Kills the pop before inertial blending lands.

Directional strafe set

Strafe clips declared, never existed as assets. Claude bpy derives from walk clip: rotate pelvis + shoulders 90°, adjust foot phase, bake. Zero human animation time.

NLA layer setup

Configures NLA layer order: base loco → upper body override → additive. Exports GLB with tracks matching alias map. Drop-in replacement.

The new pipeline

Claude writes bpy script → Blender executes it → NLA strips created
→ glTF Export (Draco compressed) → public/models/*.glb
→ assetManager.register() → enemyGLTF.js resolves aliases
→ buildEnemyMixer() pre-loads all actions → game loop
Why this matters for a solo project

Pro animation = mocap money or animator time. Claude bpy scripts = procedural keyframes at conversation speed. Quality ceiling below handcrafted — but for breathing, hit reactions, additive layers: sufficient. Always refinable in Blender after.
