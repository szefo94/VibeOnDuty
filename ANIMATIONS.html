The full call chain

Every animation decision in the game flows through this path, once per entity per frame:

requestAnimationFrame — loop.js
  └─ updateEnemies(ts, dt)
     └─ e._aiState.tick(e, dt, ctx) → isMoving
     └─ tickEnemyAnimation(e, dt, isMoving)
         ├─ jump phase state machine (start / loop / land)
         ├─ clip selection: boolean waterfall
         └─ crossfade(e, clip, snapTransition ? 0 : 0.22)
             └─ mixer.update(dt)

What actually fires — and what doesn't

The alias table in enemyGLTF.js defines 20 clip names. Here is the honest runtime picture:

Machine observation

70% of the declared clip vocabulary is dead code at runtime. The clips exist in the alias table and load correctly from GLB assets, but the clip-selection waterfall in tickEnemyAnimation never reaches them. This is not a missing-asset problem — it is a missing-routing problem.

How locomotion is currently decided

Speed in this system is not a continuous value — it is a boolean. The enemy is either moving or not. If it is moving in attack state, it plays run. If not, it plays attack. There is no interpolation between these states, no awareness of actual velocity magnitude, and no directionality.

// Current selection logic (simplified) — enemyAnimations.js if (state === 'attack' || state === 'spotted') { clip = isMoving ? 'run' : (muzzleFlashT > 0 ? 'shoot' : 'attack'); } // isMoving = velocity.length() > 0 — there is no speed value, only on/off // result: a soldier going 0.1 m/s and 6 m/s play identical animations

The crossfade problem

Three.js's crossFadeTo() is avoided in the codebase because it freezes the incoming clip at frame 0. The workaround — fadeOut + reset().fadeIn().play() — is correct and solves that bug. But the crossfade duration is a flat constant: 0.22s for smooth transitions, 0s for snap clips.

This means a transition from sprinting full-speed into a crouch gets the same 0s snap as a bullet-to-death transition. It also means a slow shuffle into idle gets the same 0.22s blend as a sudden directional reversal. There is no concept of carry-over pose, velocity, or context.

Part II

Where It Breaks

These are the specific failure modes a player sees, mapped back to their root causes in code. Ordered by impact on perceived quality.

  • 01
    Walk and run are parallel universes

    A soldier going 0.5 m/s and one going 6 m/s play the same run clip. There is no speed-parameterized blend. The result is foot-sliding at low speeds and a slow-motion feeling at high speeds because stride rate never matches actual velocity.

    High
  • 02
    Strafe is invisible

    Enemies strafe perpendicular to the player in attack state (the velocity is applied in enemyStates.js). They slide sideways while playing the forward run clip. The strafe clips are declared and loaded but no code path selects them.

    High
  • 03
    Shoot animation is sub-perceptual

    The shoot clip fires only while muzzleFlashT > 0, a window of ~55ms. Human visual persistence is ~100–150ms. The clip starts, gets immediately crossfaded away by the next frame's state, and the player perceives no shooting animation at all — only a muzzle flash.

    High
  • 04
    No additive layer — everything is a full-weight replacement

    Three.js AnimationMixer supports additive blending (THREE.AdditiveAnimationBlendMode), but it is unused. Every clip competes for 100% of total weight. This means a hit reaction replaces locomotion rather than sitting on top of it. Breathing, aim sway, and recoil have no clip representation at all.

    High
  • 05
    Upper body and lower body are locked together

    There is no bone mask. If an enemy is walking and shooting, one clip must win. In practice, locomotion wins and the shooting arm gets no animation — the gun floats forward at a fixed pose while the legs cycle. A proper split at the spine would let the legs walk while the arms aim and reload independently.

    High
  • 06
    Crossfade ignores context — 0.22s is never the right answer

    Blending from idle to walk feels slow at 0.22s. Blending from a full sprint into a stopped aim pose feels abrupt at 0.22s. The ideal transition duration depends on the velocity delta and the semantic distance between clips. Inertial blending (carry pose momentum from the outgoing clip) makes every transition feel right automatically.

    Med
  • 07
    No procedural foot IK

    Ramp tiles raise the floor geometry. Enemies and the player walk onto ramps with feet visibly embedded in or floating above the surface. A two-bone IK pass on both ankles, raycasting per-foot to the collision surface, would ground the characters without requiring custom animation per slope angle.

    Med
  • 08
    Death has no lead-in

    killEnemy snaps to death at duration 0. This is technically correct — a dead enemy should not spend 0.22s elegantly transitioning — but without a preceding hit-stagger additive layer, the enemy goes from full animation to ragdoll-freeze in one frame, which feels like a bug rather than a death.

    Low
  • 09
    Procedural weapon bob competes with GLTF body animation

    The first-person weapon bob is a manually calculated sine offset applied each frame to wpn.position. The third-person GLTF skeleton animates independently. They are not synchronised — the weapon floats in first person while the body cycles below. A shared locomotion phase value would let both systems breathe together.

    Low
Part III

The Evolution Plan

Five changes, independent enough to ship incrementally, each closing a specific gap identified above. Listed in implementation order — each one builds on the previous.

A

Locomotion Blend Tree

locomotion no new assets needed

Replace the isMoving boolean with a continuous speed float (0–1 normalized). Blend idle → walk → run by linearly crossfading weights as speed changes, rather than hard-switching clips. This single change eliminates foot-sliding at low speeds and makes deceleration feel physical.

Simultaneously, wire strafe_l and strafe_r by computing the dot product of velocity against the entity's forward vector. Negative lateral dot → strafe_l weight, positive → strafe_r. Blend all three axes (forward/back, strafe, speed) into a 2D blend tree.

// Proposed: 2D locomotion blend (forward × strafe) const vFwd = dot(velocity, forward); // –1 … 1 const vRight = dot(velocity, right); // –1 … 1 const speed = velocity.length() / MAX_SPEED; actions.idle.weight = Math.max(0, 1 - speed * 2); actions.walk.weight = triangular(speed, 0, 0.5, 1); actions.run.weight = Math.max(0, speed - 0.5) * 2; actions.strafe_l.weight = Math.max(0, -vRight) * speed; actions.strafe_r.weight = Math.max(0, +vRight) * speed; // normalise weights so they sum to 1.0
B

Upper / Lower Body Split via Bone Mask

body masking blender asset needed

Three.js AnimationAction supports a weight-per-track model. By zeroing the weight of spine-upward tracks on the locomotion layer and zeroing leg-bone tracks on the aim layer, you get a clean split: legs walk, torso shoots.

Upper body clips (shoot, reload, attack) run on a separate action with only arm/spine/head tracks active. Lower body clips (walk, run, idle) run with leg tracks only. The Quaternius rig's named bones (spine_01, thigh_l/r) are already resolved in enemyGLTF.js for the debug skeleton — the same bone lookup enables masking.

Impact on shoot animation

With upper-body masking, the shoot clip no longer competes with locomotion for full weight. It runs independently on the arm/spine layer for the duration of the shoot animation (300–400ms), while the legs continue their correct cycle below. The 55ms muzzle-flash window is no longer the bottleneck.

C

Additive Layer Stack

additive blending new clips needed

Enable THREE.AdditiveAnimationBlendMode on a dedicated additive action group. Three types of clip live here, each at independent weight, all applied on top of the base locomotion result:

Breathing

A 3-second looping clip with subtle spine compression and shoulder rise. Weight = 1.0 always. When the player sprints, weight rises to 1.4 (heavier breathing). Costs one Blender export pass.

Hit Reaction

Short (0.3s) directional additive clips: hit_front, hit_back, hit_left, hit_right. Playing an additive hit clip on top of locomotion gives a stagger without interrupting the walk cycle. Weight lerps from 1.0 to 0 over the clip duration.

Aim Offset

An additive pose that tilts spine and arm toward the look target. Driven by the vertical look angle (pitch). No clip required — a procedural spine rotation applied additively each frame covers this case.

Recoil

Currently a procedural weapon position offset in shoot.js. Moving it to an additive clip means the body rocks back slightly on fire, not just the camera. The existing recoilT float drives the clip weight.

D

Inertial Blending

transitions no new assets

Inertial blending, described by David Bollo (Ubisoft) and adopted in many AAA engines, records the current pose and its first derivative (velocity) at the moment a transition fires, then decays toward the incoming clip using a critically-damped spring. The result: transitions always feel right regardless of duration, because the blend curve is shaped by physics, not a fixed lerp.

A minimal implementation in Three.js stores the outgoing bone quaternions and angular velocities at crossfade time, then applies a spring-damper correction offset that decays to zero over ~0.15s. The spring constant controls how snappy or floaty the settle feels.

// Inertial transition sketch function onTransition(bone, currentQ, nextQ) { const offset = currentQ.clone().multiply(nextQ.conjugate()); const vel = angularVelocity[bone.uuid] ?? new THREE.Quaternion(); inertia[bone.uuid] = { offset, vel, t: 0 }; } function tickInertia(bone, dt) { const s = inertia[bone.uuid]; if (!s) return; // critically-damped spring: x'' = –ω²x – 2ωx' s.t += dt * OMEGA; const decay = (1 + s.t) * Math.exp(-s.t); bone.quaternion.premultiply( slerpQ(identityQ, s.offset, decay) ); }
E

Procedural Foot IK

IK terrain-aware

Two-bone IK on each ankle (thigh → calf → foot). Each frame, raycast downward from the foot's animated world position. If the hit point is above the animated foot position, solve IK to plant the foot at the hit. If below, leave the animated pose.

An additional hip offset lowers the pelvis when both feet need to reach downward (walking into a depression) and raises it when ascending a ramp. The result is characters that feel grounded on ramps, cracks, and column bases without per-terrain animation work.

The collision surface is already queryable — the same BVH that handles player movement reports height per cell. A per-foot raycast against wallMeshes (already in the scene) returns the exact surface normal needed for ankle rotation.

Scope note

Foot IK is the most technically complex step. It should ship last, after steps A–D are stable. The visual gain on flat maps is zero; the gain on ramp-heavy maps like Vanguard Complex is significant.

Part IV

Claude × Blender

Anthropic has released native Blender integration for Claude — Claude can now read, write, and execute bpy (Blender Python API) scripts directly inside the Blender environment. This changes the asset pipeline for this project significantly.

Instead of hand-animating clips in Blender, Claude can generate a Python script that builds the animation programmatically — bone by bone, frame by frame — and bakes it into NLA strips ready for GLB export. The artist's job shifts from keyframing to curating and tuning the output.

What Claude can generate for this project

Additive clip set

A bpy script that creates breathing, hit_front, hit_back, hit_left, hit_right as additive NLA strips on the Quaternius rig. Each clip is generated from procedural keyframes (spine compression curves, arm flinch arcs) and exported with correct ANIM_RETARGET correction so they slot into the existing alias map without changes.

Locomotion polish pass

A script that reads the existing walk/run clips and generates smooth transition poses — the frames between walk and run that currently don't exist. Output: a blended "walk_to_run" and "run_to_walk" clip of 6 frames each, eliminating the pop even before inertial blending is implemented.

Directional strafe set

Strafe clips are declared in the alias map but never existed as real assets. A Claude-written bpy script derives them from the existing walk clip by rotating the pelvis and shoulder orientation 90°, adjusting foot phase, and baking the result. No human animation time required.

NLA layer setup

A script that configures the Blender NLA editor with the correct layer order: base locomotion, upper body override, additive effects. Exports a single GLB with all tracks named to match the existing alias map. Drop-in replacement for the current asset.

The new pipeline

Claude writes bpy script → Blender executes it → NLA strips created
→ glTF Export (Draco compressed) → public/models/*.glb
→ assetManager.register() → enemyGLTF.js resolves aliases
→ buildEnemyMixer() pre-loads all actions → game loop
Why this matters for a solo project

Professional 3D animation requires either expensive motion capture or expensive animator time. Claude generating bpy scripts makes procedural keyframe generation available at conversation speed. The quality ceiling is lower than handcrafted motion, but for secondary clips (breathing, hit reactions, additive layers) it is more than sufficient — and the clips it generates can always be manually refined in Blender afterward.
